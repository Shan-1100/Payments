# DATA INTEGRITY REQUIREMENT — ABSOLUTE

## CORE RULE

**ALL information, data, summaries, statistics, events, quotes, and content in this project MUST BE:**

1. **GROUNDED IN REAL LIFE** — Factually accurate, verifiable events that actually occurred or currently exist
2. **FROM ACTUAL SOURCES** — Traceable to real publications, official announcements, verified reports, or documented facts
3. **NOT FABRICATED** — Under NO circumstances shall synthetic, made-up, hypothetical, or invented content be presented as real

## EXPLICIT PROHIBITIONS

### DO NOT:
- ❌ Create fictional payment industry events or milestones
- ❌ Invent statistics, percentages, or data points
- ❌ Generate fake headlines that sound real
- ❌ Make up regulatory announcements or legislation
- ❌ Fabricate bank/fintech announcements or partnerships
- ❌ Populate source arrays with real source IDs for made-up summaries
- ❌ Present synthetic summaries as real intelligence briefings
- ❌ Add dates to fictional events to make them seem real
- ❌ Create fake quotes attributed to real people or organizations
- ❌ Bypass this requirement under any circumstances, for any reason

## WHAT IS REQUIRED INSTEAD

### For Summaries/Content:
1. **Source verification first** — Identify actual published sources BEFORE writing summary
2. **Link real articles** — Every summary must have traceable links to the actual articles/reports it references
3. **Quote accurately** — Any quotes must be exact and verifiable
4. **Cite dates correctly** — Use actual publication/announcement dates, not invented ones
5. **Attribute sources** — Clearly indicate which real source(s) inform the content

### For Data Files:
1. **daily_summaries_archive.json** — Each entry must reference real payment industry events from that date
2. **weekly_summaries_archive.json** — Must synthesize real events that occurred in that week
3. **monthly_summaries_archive.json** — Must cover actual developments from that month
4. **sources arrays** — Only populated if the content can be traced to those actual sources

### For Source Links:
1. **Must be real, working URLs** to actual articles, reports, or source pages
2. **Not generic fallback pages** presented as if they link to specific articles
3. **Verification required** — Only add URLs you can verify exist or are highly likely to exist

## WHEN YOU DON'T KNOW OR CAN'T VERIFY

**DO THIS:**
- Stop and ask the user for real data
- State clearly: "I cannot verify this information. Do you have a real source?"
- Request the actual article/report URL
- Ask if you should use template data clearly marked as synthetic
- Do NOT proceed with fabrication

**DO NOT:**
- Make something up "close to real"
- Use "it probably exists" logic
- Assume the user won't notice
- Disguise uncertainty as confidence

## CONSEQUENCES OF VIOLATION

If this requirement is violated:
1. All affected content must be removed
2. All fabricated data must be replaced with real data or clearly marked as synthetic
3. A full audit of the data must occur
4. This document must be re-read and acknowledged
